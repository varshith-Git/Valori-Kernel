# Valori Desktop (Tauri)

The native control-plane shell — "Docker Desktop for AI Memory." See
[`rfcs/0006-desktop-daemon-architecture.md`](../rfcs/0006-desktop-daemon-architecture.md)
for the architecture this is building toward.

**Status (Phase D1.3): a real, installable desktop app.** `ui/` (the existing
Next.js app) is the canonical frontend — there is no separate desktop React
app, and there won't be one. `tauri build` produces a genuine, working
installer (`.dmg` verified on macOS; `.msi`/`.AppImage` built by CI — see
`.github/workflows/desktop-build.yml`) that needs nothing on the end user's
machine: no Rust, no Node.js, no terminal. Native Rust owns the
`valori-daemon` process lifecycle (D3) — the desktop launches it, points it
at the user's chosen workspace, and shuts it down gracefully on quit (window
close *or* an external SIGTERM — see `src/lib.rs`) — and everything is
bundled: `valori-daemon` + `valori-node` as Tauri sidecars (D3.1), `ui/`
itself as a bundled Node sidecar + standalone server resource (D1.3, since
`ui/` can't statically export yet). See
[`docs/architecture/desktop-layout.md`](../docs/architecture/desktop-layout.md)
for the full bundle layout and
[`docs/DESKTOP_RELEASE_CHECKLIST.md`](../docs/DESKTOP_RELEASE_CHECKLIST.md)
for the manual smoke test to run before calling a build shippable.

> Decision record: an earlier pass briefly built a second, from-scratch
> frontend under `desktop/src` (Vite + React) to avoid `ui/`'s server-side API
> routes blocking static export. That was reverted — `ui/` already implements
> the full product surface (projects, collections, search, GraphRAG, audit,
> metrics, cluster, settings, theming, etc.); rebuilding it in a second stack
> would mean maintaining two frontends. The correct fix is migrating `ui/`'s
> project/workspace/collection/model **lifecycle** logic
> (`src/lib/server/process-manager.ts`, `projects.ts`) onto `valori-daemon`,
> then making `ui/` static-exportable. Data-operation routes (search, insert,
> GraphRAG, audit, etc.) stay exactly as they are, talking directly to the
> running `valori-node` — the daemon is not a proxy for those.

## Layout

```
desktop/
  package.json            # tauri CLI + scripts (dev / build / icon)
  scripts/
    prepare-sidecars.mjs  # resolves host target triple, builds/locates valori-daemon
                          # + valori-node + copies the Node runtime, into
                          # src-tauri/binaries/ (D3.1)
    prepare-ui-server.mjs # release-only: `next build` (standalone) + copies
                          # the result into src-tauri/resources/ui-server/ (D1.3)
    dev.mjs               # beforeDevCommand: preps sidecars, then starts ui/'s dev server
  src-tauri/
    Cargo.toml            # standalone Rust crate (NOT in the root workspace)
    tauri.conf.json       # window + dev server (points at ui/ on :3001);
                          # externalBin + resources (D3.1/D1.3)
    build.rs
    loading/              # tiny static "Starting Valori…" page — the real
                          # frontendDist in release builds (D1.3)
    binaries/             # generated per-platform by prepare-sidecars.mjs — gitignored
    resources/            # generated by prepare-ui-server.mjs — gitignored
    src/
      main.rs             # thin entry → lib::run()
      lib.rs              # Tauri app + node_health command; wires daemon_manager +
                           # ui_server_manager; graceful shutdown (window close AND
                           # SIGTERM), guarded against re-entrant AppHandle::exit() (D1.3)
      daemon_manager.rs    # owns the valori-daemon child process: start_daemon /
                           # stop_daemon / daemon_status Tauri commands (D3);
                           # dev target-dir search vs. release sidecar spawn (D3.1)
      ui_server_manager.rs # release-only: spawns the bundled ui-server + navigates
                           # the main window to it once healthy (D1.3)
    capabilities/
      default.json        # Tauri 2 permissions
    icons/                # generated via `npm run icon`, committed
```

The Rust crate is deliberately outside the root Cargo workspace (its own empty
`[workspace]` table + `exclude` in the root `Cargo.toml`), so `cargo build
--workspace` and every backend CI job are unaffected.

## Prerequisites

- Rust (already required by this repo) + the Tauri system deps for your OS
  (macOS: nothing extra — WebKit is built in; Linux: `webkit2gtk`; Windows:
  WebView2).
- Node ≥ 18 (for `ui/`).

## First-time setup

```bash
cd desktop
npm install                      # installs @tauri-apps/cli + api
```

Icons are already generated and committed under `src-tauri/icons/`. To
regenerate from a new logo: `npm run icon <path-to-logo.png>`.

## Run (dev)

```bash
cd desktop
npm run dev            # = tauri dev
```

`tauri dev` runs `beforeDevCommand` (`node scripts/dev.mjs`), which first
preps the sidecar binaries (`scripts/prepare-sidecars.mjs`, dev mode — reuses
whatever's already built under the root workspace's `target/`, only building
if nothing exists) and then starts the Next.js UI on `:3001`, then opens a
native window pointed at it. The dev-mode sidecar prep step is required even
though dev mode doesn't actually spawn sidecars (it uses the `target/{release,debug}`
search path instead) — Tauri's build script validates `externalBin` resource
paths on every cargo build, not just `tauri build`. The desktop launches
`valori-daemon` itself once a workspace is chosen (Welcome flow) or on every
launch for returning users; you don't need to run it by hand.

## Calling native code from the web UI

```ts
import { invoke } from "@tauri-apps/api/core";
const report = await invoke("node_health", { baseUrl: "http://127.0.0.1:3000" });
// → { url, reachable, body }
```

**Daemon lifecycle (D3)** — exposed via `ui/src/lib/native.ts`
(`startDaemon`/`stopDaemon`/`daemonStatus`), no-ops outside the desktop shell:

```ts
import { startDaemon, stopDaemon, daemonStatus } from "@/lib/native";

await startDaemon("/Users/me/valori-workspace"); // spawns valori-daemon with VALORI_HOME=<home>, waits for /health
await daemonStatus();                            // { running, healthy, bind }
await stopDaemon();                               // POST /v1/shutdown, waits for graceful exit
```

`Welcome.tsx`'s `finish()` calls `startDaemon(workspaceDir)` right after the
user picks a workspace folder — that's what makes the folder picker actually
control where data lives, instead of being cosmetic. `AppShellGate` calls it
again on every launch for returning users, using the persisted
`workspaceDir` preference. The app's `RunEvent::ExitRequested` handler calls
the equivalent of `stop_daemon` before letting the window close, so every
running project gets a snapshot before the daemon process exits.

This is the seam where the desktop becomes a control plane: native Rust owns
process/daemon lifecycle; the web UI stays a pure view.

## Production build

```bash
cd desktop
npm run build           # = tauri build
```

Runs `beforeBuildCommand` (`prepare-sidecars.mjs --release` — rebuilds
`valori-daemon`/`valori-node` in release mode and copies the Node runtime;
then `prepare-ui-server.mjs` — runs `next build` in `ui/` and packages the
standalone output), then bundles everything into the platform's native
installer. Output lands under
`src-tauri/target/release/bundle/{macos,dmg,msi,appimage}/`.

This works today (verified: real `.dmg` produced, `hdiutil verify`-clean,
launched and confirmed the bundled sidecars actually start and serve the
app) even though `ui/` still can't statically export — see
[`docs/architecture/desktop-layout.md`](../docs/architecture/desktop-layout.md)
for why (`ui/` is bundled as a Node sidecar + standalone-server resource,
not embedded as static assets). Per RFC-0006, migrating `ui/`'s remaining
lifecycle routes onto the daemon and switching to `output: "export"` is
still the long-term direction (would drop the Node sidecar entirely), but
it's no longer a blocker for shipping.

CI builds all three platforms on every push touching `desktop/`,
`crates/valori-daemon`, `crates/valori-node`, or `ui/`
(`.github/workflows/desktop-build.yml`) and uploads each installer as an
artifact. Code signing and notarization are not done yet (Phase D1.4) — all
current builds are unsigned.

**Before calling a build shippable**, run through
[`docs/DESKTOP_RELEASE_CHECKLIST.md`](../docs/DESKTOP_RELEASE_CHECKLIST.md) —
install → launch → Welcome → create a project/collection → insert/search →
quit → relaunch → confirm the data survived. That's a manual checklist on
purpose; it hasn't been automated yet.

### A gotcha if a build fails partway through

If `tauri build` fails during DMG bundling on macOS, check for a stale
mounted disk image left by the failed attempt (`hdiutil info`) — each failed
run can leave a `/Volumes/dmg.XXXX` mount behind, and the *next* attempt's
failure looks unrelated until you trace it back. `hdiutil detach` any stale
`dmg.*` volumes, then retry.
