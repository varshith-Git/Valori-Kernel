# Valori Kernel: Module Analysis - State Machine & Event Sourcing

This is the third module breakdown. We have seen how Math, Storage, and Graphs work in isolation. The **State Machine & Events** module (`src/state/`, `src/event.rs`) is the "brain" that combines them all. 

Valori is an **Event-Sourced** database. This means its state is not merely a collection of files that change over time; its state is strictly the result of applying a sequence of immutable `Commands` (Events) to an empty state.

---

## 1. The Core Struct: `KernelState`

**Location**: `src/state/kernel.rs`

### `KernelState`
```rust
pub struct KernelState {
    pub dim: Option<usize>,
    pub(crate) version: Version,
    pub(crate) records: RecordPool,
    pub(crate) nodes: NodePool,
    pub(crate) edges: EdgePool,
    pub(crate) index: BruteForceIndex,
}
```
This is the god-object of the deterministic engine. It holds:
- **`dim`**: The vector dimension (auto-detected on the very first insertion).
- **`version`**: A monotonically increasing integer tracking how many state transitions have occurred.
- **The Pools**: Instances of the `RecordPool`, `NodePool`, and `EdgePool`.
- **The Index**: A `BruteForceIndex` for executing search queries across the records.

## 2. The Language of Mutation: Commands & Events

**Location**: `src/state/command.rs` and `src/event.rs`

To modify the `KernelState`, external code *cannot* touch the pools directly. It must formulate a `Command` or `KernelEvent`.

### `Command` / `KernelEvent`
```rust
pub enum Command {
    InsertRecord { id: RecordId, vector: FxpVector, metadata: Option<Vec<u8>>, tag: u64 },
    DeleteRecord { id: RecordId },
    CreateNode { node_id: NodeId, kind: NodeKind, record: Option<RecordId> },
    CreateEdge { edge_id: EdgeId, kind: EdgeKind, from: NodeId, to: NodeId },
    DeleteNode { node_id: NodeId },
    DeleteEdge { edge_id: EdgeId },
}
```
- **Serialization**: `KernelEvent` overrides `bincode` serialization explicitly. This guarantees that an event recorded today will decode perfectly years from now, preserving the audit trail.
- **Why Both?**: `KernelEvent` is the durable format stored in the `.log` WAL, and `Command` is the runtime execution primitive inside the engine. `KernelState::apply_event` seamlessly translates the former to the latter.

## 3. Execution: The `apply` Function

**Location**: `src/state/kernel.rs` (`apply` and `apply_event`)

When an event enters the system via the Python client, it hits `KernelState::apply(&mut self, cmd: &Command) -> Result<()>`.

### Step-by-Step Logic
1. **Dimension Locking (`InsertRecord`)**:
   - If `self.dim` is `None`, it permanently locks the engine to `vector.len()`.
   - If `vector.len()` doesn't match `self.dim`, it throws `KernelError::InvalidOperation`.
2. **Metadata Checks**:
   - Ensures `metadata.len()` stays under `MAX_METADATA_SIZE` (64KB default limit) to prevent memory ballooning in `no_std`.
3. **Execution & Verification**:
   - For `CreateEdge`, it calls `add_edge(...)` on the pools. 
   - *Crucial Check*: It verifies that the internal array index generated by the pool `allocated_id` exactly matches the `id` requested by the `Command`. If they mismatch, the system halts. This enforces that memory is perfectly synchronized with the event log.
4. **State Advancement**:
   - Ends with `self.version = self.version.next()`.

## 4. Complex Mutability: Cascading Deletes

**Location**: `src/state/kernel.rs` (`_delete_node` and `_delete_edge`)

Deleting a Record or an Edge is simple. Deleting a **Node** is highly complex because it has edges attached to it.

### `_delete_node(node_id: NodeId)`
1. Scans the entire `EdgePool` to find any edge where `from == node_id` OR `to == node_id`.
2. For each attached edge, it calls `_delete_edge(edge_id)`.
3. Once all relationships are severed, the node itself is deleted from the `NodePool`.

### `_delete_edge(edge_id: EdgeId)` (Linked List Unlinking)
When an edge is deleted, the adjacency linked list must be repaired:
1. It looks up the `from` node of the edge.
2. It walks the `next_out` linked list from the head (`first_out_edge`).
3. If the edge being deleted is the **Head**, it updates the Node's `first_out_edge` to point to `edge.next_out`.
4. If it is in the **Interior**, it updates the `prev_edge.next_out` to jump over the deleted edge and point to `edge.next_out`.
5. Finally, it removes the edge from the `EdgePool`.

## 5. Defense in Depth: Invariant Checking

**Location**: `src/state/kernel.rs` (`check_invariants`)

Because this engine acts as a source of truth for cryptographic proofs, `check_invariants` is available to panic the system during tests or shadow execution if memory becomes corrupted:
- **Index Alignment**: Checks `node.id.0 == array_index`.
- **Dangling Pointers**: Ensures every `Node->Record`, `Node->Edge`, and `Edge->Node` points to a `Some()` object in the physical pool.
- **List Coherence**: Ensures `next_out` edges actually belong to the correct `from` node.

If a Command causes any of these to fail during the shadow apply step, the event is rejected and never hits the live state.
